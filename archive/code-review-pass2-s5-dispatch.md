**Proposal:** Add a single type-only per-capability payload/reply map to transport.ts (the seam both tiers already import types from), and derive PageCapability from it:

```ts
// transport.ts — types only, no runtime additions
import type { PencilParams, PenSample, Tool } from "@habemus-papadum/aiui-pencil";
import type { AppSelection } from "@habemus-papadum/aiui-lowering-pipeline";

export type Ack = { ok: true };
export type CapError = { error: string };

export type PencilOp =
  | { op: "size" }
  | { op: "engage"; fadeSec: number }
  | { op: "disengage" }
  | { op: "fade"; fadeSec: number }
  | { op: "clear" }
  | { op: "undo" }
  | { op: "rbegin"; id: string; init: { tool: Tool; params: PencilParams; point: PenSample } }
  | { op: "rpoint"; id: string; point: PenSample }
  | { op: "rend"; id: string; point?: PenSample }
  | { op: "rcancel"; id: string };

export interface PageCapabilityMap {
  heartbeat: { payload: { session: string }; reply: Ack };
  ring:      { payload: PageRing; reply: Ack };                       // ← folds the "ring" hole in
  flash:     { payload: { kind: "shot" | "miss" }; reply: Ack };
  keylayer:  { payload: { capture: boolean }; reply: Ack };
  selection: { payload?: undefined; reply: (AppSelection & { title?: string }) | null };
  viewport:  { payload?: { sample: boolean; mode: string }; reply: Ack | { w: number; h: number; dpr: number } };
  region:    { payload: { arm: boolean }; reply: Ack };
  jump:      { payload: { arm: boolean }; reply: Ack | CapError };
  pencil:    { payload: PencilOp; reply: Ack | CapError | { width: number; height: number } };
  toolsCall: { payload: { ns: string; name: string; args?: unknown; callId: string }; reply: Ack };
}
export type PageCapability = keyof PageCapabilityMap;                 // replaces the string union

export interface PageTransport {
  requestPage<C extends PageCapability>(
    tab: number, capability: C, payload?: PageCapabilityMap[C]["payload"],
  ): Promise<PageCapabilityMap[C]["reply"] | undefined>;              // undefined = unreachable tab, today's contract
  broadcastRing(state: RingState): void;
  onPageEvent(handler: (event: PageEvent) => void): () => void;
}
```

Sharing across both tiers — the load-bearing fact, stated explicitly: page-script.ts's pageBootstrap is STRINGIFIED for injection (buildPageScript, page-script.ts:666-670) and may reference no imported VALUE inside its body — but `import type` declarations ERASE at transpile time, and `pageBootstrap.toString()` runs on the already-transpiled function, so type-only imports (and inline casts/`satisfies` in the body) leave zero runtime trace. page-script.ts already imports the pageTabRecord VALUE safely because it only stringifies it outside the body; the same file can add `import type { PageCapabilityMap, PencilOp } from "../transport"` and type its `handle` as `<C extends PageCapability>(capability: C | string, payload) => PageCapabilityMap[C]["reply"] | CapError`, with each case's return checked via `satisfies`. What CANNOT be shared into page-script is runtime decoders/predicates — so handler-side defensive narrowing stays inline there (and should: the wire crosses JSON/structured-clone boundaries from possibly-stale senders — the frozen old extension, an orphaned content script — so compile-time types must not replace the `=== true` / `String(...)` coercions, only anchor them to one declared shape per capability instead of ten scattered ad-hoc casts).

What each cast becomes:
- claims.ts:52-163, lanes.ts:294/456/476/488/495/564, pencil-host.ts:81-117, tools-link.ts:117 — sender payloads typecheck against the map; pencil-host's `forward(payload: Record<string, unknown>)` becomes `forward(op: PencilOp)`; its size-reply cast (pencil-host.ts:119) and extension-bus's viewport-reply cast (extension-bus.ts:338) become typed replies.
- lanes.ts:481 `engine.appSelection(selection as never)` — dies once the selection reply IS `AppSelection & { title?: string }` (AppSelection already lives in aiui-lowering-pipeline, an allowed dependency of the intent client).
- content.ts: replace `serveRelay(PAGE_ADDRESS, {...})` handlers' ten casts with one typed wrapper `servePageCapabilities(handlers: { [C in PageCapability]: (payload: unknown) => PageCapabilityMap[C]["reply"] } & { driverGone: () => Ack })` — serveRelay itself stays `Record<string, RelayHandler>` (the sw.ts BROKER_ADDRESS table also uses it); only the PAGE_ADDRESS table gets the typed shell. Each handler keeps its runtime narrowing but its RETURN is now checked, which is where tier drift would surface at compile time.
- content.ts:453/456/459 `p.init as never` / `p.point as never` — become one narrowing to PencilOp, then typed calls into PencilHandle (whose signatures at page/pencil-mount.ts:74-77 already match the map's rbegin/rpoint/rend shapes).
- cdp-bus.ts:248/270 `PageCapability | "ring"` — collapses to PageCapability once ring is in the map; the sticky map (cdp-bus.ts:147) and ext sticky (extension-bus.ts:69) key on the derived union unchanged.
- driverGone stays OUT of PageCapabilityMap deliberately: it is not a panel capability (the service worker sends it, only the MV3 tier serves it, requestPage never carries it). It belongs beside the map as an ext-protocol extension: declare it in ext/protocol.ts as `type ExtOnlyCommand = "driverGone"` with reply Ack, consumed by sw.ts:175 and the servePageCapabilities wrapper. This keeps the shared map an honest both-tiers contract.
- The ext MAIN-world postMessage hop (content.ts:260-268, content-main.ts) reuses `PageCapabilityMap["toolsCall"]["payload"]` and the toolsResult shape from PageReport for its field-tagged messages — the envelope stays ad-hoc, the payloads stop being re-declared.

PageReport/PageEvent (the reverse wire) are already typed unions shared by both tiers (page-script.ts:32-69, transport.ts:93-147) and need no change; the map only closes the request direction.

**Risks:** 1) Tier divergence is real, not incidental: `selection` and `viewport` genuinely reply DIFFERENT shapes per tier (CDP selection is text-only, page-script.ts:528-534, vs ext structured, content.ts:387-403; CDP viewport is {ok:true} vs ext {w,h,dpr}). The proposed union types document this drift; they do not fix it. Narrowing the unions later (e.g. making CDP selections structured) is a behavior change, out of scope for this refactor — but a reviewer should not mistake the union for an endorsement. 2) Compile-time types cannot validate the wire: payloads cross JSON (Runtime.evaluate stringify) and chrome structured-clone boundaries, and stale speakers exist by design (the FROZEN old extension shares the `aiui:1` relay format per ext/relay.ts's header; an orphaned content script survives ext:watch). Handler-side defensive coercions must stay; the refactor moves casts to one declared shape per capability, it does not delete runtime narrowing. Removing the coercions "because it's typed now" would be the actual regression. 3) The stringified page-script constraint is easy to silently break later: someone adding a shared runtime validator and importing it into pageBootstrap's body would inject a ReferenceError into every driven page. The contract must be stated in page-script.ts's header (type-only imports OK, value imports never) and ideally guarded by the existing fingerprint/typecheck discipline — note that a types-only edit does NOT change pageBootstrap.toString(), so it does not bust the injected version; already-open documents keep serving the old runtime behavior while the panel compiles against new types (fine for erasable types, a trap if anyone assumes redeploy). 4) `viewport`'s `{sample, mode}` payload is exercised only by the videoSample claim's DEFAULT applier (claims.ts:132-150) — a fake-host/test path; both real handlers ignore it. Typing it fossilizes a vestigial shape; consider annotating it deprecated in the map rather than widening tests around it. 5) `flash`'s "miss" kind is handler-supported but never sent — same class of decision (keep in the union as declared vocabulary, or prune; pruning touches page behavior). 6) Doc drift to fix in passing, not in the map: "locate" appears in the capability lists of page-script.ts:12 and ext/protocol.ts:14 but is not a served capability (locateComponents is internal to region handling). 7) Dependency check: `import type` of PenSample/PencilParams/Tool from aiui-pencil and AppSelection from aiui-lowering-pipeline adds no new workspace edges (aiui-intent-client already depends on both at runtime) and respects the stated direction (client → runtime + lowering + viz); nothing here touches aiui-viz, the channel, vscode, or the create-aiui template. 8) Making serveRelay itself generic would break the sw.ts broker table (sw.ts:212-229, its own casts are out of this wire's scope) — hence the wrapper-only approach; a reviewer proposing "just type serveRelay" should check BROKER_ADDRESS first. 9) The `title` field in both tiers' selection replies is not in AppSelection — today the `as never` cast hides it, and engine.appSelection ignores it. Typing the reply as `AppSelection & { title?: string }` preserves the wire bytes; alternatively adding `title` to AppSelection touches the lowering pipeline's event schema (and anything that serializes it) — bigger blast radius, not needed for cast removal.

### TraceStageEvent live tap (trace.ts:64-80 onStage sink) → `serve`'s terminal narration: a THIRD label matcher, prefix/equality based, in the channel itself

| op | payload | senders | handlers |
| --- | --- | --- | --- |
| `isNarratedTraceStage membership: startsWith('linter'), startsWith('transcription'), startsWith('cost:'), === 'composed intent', === 'fin compose'` | label string only (narration prints `${threadId} ${label}`) | packages/aiui-claude-channel/src/trace.ts:280-286 (onStage dispatch in record); all wire-1 sender sites above | packages/aiui-claude-channel/src/commands/serve.ts:158-166 (definition); packages/aiui-claude-channel/src/commands/serve.ts:365-366 (use); DEAD BRANCH: startsWith('transcription') matches no current writer — transcription spend is labelled 'cost: realtime transcription seg_N' and hits the cost: branch; the old 'transcription failed' label has no writer |

**Proposal:** Add one shared, dependency-legal contract module — `trace-stages.ts` in `@habemus-papadum/aiui-lowering-pipeline` — and make every matcher a switch over ONE total parser instead of three independent string ladders. Lowering-pipeline is the only legal home: it is a leaf both current parties already import (channel: packages/aiui-claude-channel/package.json deps; trace-ui already imports IntentEvent/PromptSpan from it at trace-cards.ts:22), so no new dependency edges; aiui-build-config is build-config-only and never published, so it cannot be a runtime dep of two published packages.

The contract, three layers:

(1) A discriminated `ParsedStage` union — one variant per row of the wire-1 catalog, parameterized where the label is templated: `{ t: \"stt-partial\", segment: number } | { t: \"cost\", what: string } | { t: \"attachment\", id: string, media: \"shot\" | \"seg\" } | { t: \"input-frame\", n: number, chunk?: ChunkKind, id?: string, fin: boolean } | … | { t: \"legacy-live-tool-call\" } | … | { t: \"unknown\" }`. Legacy reader-only labels (correction trio, live *, video vid_*, voice reply, transcription failed) get explicit variants marked legacy — they are part of the persisted format forever.

(2) Paired label builders and a TOTAL parser generated from one table, so they cannot drift: `stageLabel.sttPartial(3)` returns exactly \"stt partial seg_3\"; `parseStageLabel(label: string): ParsedStage` returns `{ t: \"unknown\" }` for anything unrecognized (never throws, never omits). A round-trip test (`parse(build(x))` ≡ x) plus a golden-literal test pinning every builder output char-for-char to today's strings — the strings ARE the on-disk format, so old traces keep parsing and new traces keep rendering in old viewers. Ladder precedence quirks (e.g. `frame N attachment shot_` before bare `frame N`, `(fin)` detection) are encoded once in the parser table in the current match order.

(3) Per-variant payload TYPES used asymmetrically: the channel's writers use them as compile-time contracts (`data: {...} satisfies SttFinalData`, builders at intent-v1.ts/tracing.ts/linter-sidecar.ts call sites swapped one-for-one — byte-identical output, behavior-preserving); readers get them only through safe accessor helpers that treat data as unknown-with-guards, preserving trace-ui's deliberate degrade-don't-crash stance toward old/foreign manifests (TraceStageLike stays all-optional, sources.ts:13-19).

Consumer rewiring: classifyStage (trace-cards.ts:168-442) becomes `switch (parsed.t)` — exhaustive via a `never` check over known variants, with the `unknown` arm reproducing today's kind-keyed generic fallback (trace-cards.ts:432-441), so totality over unknown labels is preserved by construction. The parse result is stored on the card (`TraceCard.parsed: ParsedStage`), which deletes the second ladder entirely: renderCardBody (trace-view.ts:442-680) switches on `card.parsed.t` and receives typed params (segment ordinals, attachment ids) instead of re-running regexes. The scattered extractors (traceOutcome trace-cards.ts:501/507, heroPrompt/speculativePromptText/loweredPromptSpans :565-666, isPartialLabel :589) become lookups on parsed variants. serve.ts's isNarratedTraceStage (:158-166) becomes `NARRATED: ReadonlySet<ParsedStage[\"t\"]>` membership over the same parser — transcribed from CURRENT behavior (linter*, cost, composed-intent, fin-compose; the dead transcription branch dropped only as an explicitly-flagged decision). The stt diagnostic sub-dispatch (RealtimeDiagnostic.kind interpolated into labels at intent-v1.ts:926-930) gets its own variants so the diagnostic union (realtime.ts:106-110) and the label vocabulary are tied at the type level.

Optional phase 2 (additive, separate decision): stamp `TraceStage.t = parsed.t` (+ params object) into new manifests at record time; readers prefer `t` when present and fall back to label parsing. Old traces lack the field and old viewers ignore it — purely additive, and it makes any future label-wording change safe.

**Risks:** 1) The labels are a persisted wire format, not display strings: .aiui-cache traces already on disk must render forever, so the refactor must be char-for-char (golden tests against today's literals; existing oracles: trace-cards.test.ts, intent-v1.test.ts:509 which finds stages by exact label). Any \"tidying\" of a label string during the move silently orphans history.
2) Behavior-preservation vs. latent drift bugs: the catalog surfaced five — trace-view.ts:562 dead `case \"realtime commit\"` (writer suffixes ` seg_N`, intent-v1.ts:987); trace-view.ts:472 reads `data.corrector` which the writer never sends (intent-v1.ts:581-598) so 'fix ?' always; trace-view.ts:656-658 condition body reads engaged/enabled but the writer sends {identity:true} (intent-v1.ts:1156) so 'off' always; serve.ts:161 dead startsWith('transcription') narration branch; and unclassified live labels ('linter control' intent-v1.ts:1214, 'realtime discard seg_N' :1020, 'user text' processors.ts:67) falling to generic cards. A strictly behavior-preserving pass must REPRODUCE these; each fix is a separate, visible-behavior decision the document should enumerate.
3) Precedence and coalescing are behavior: the current ladder is order-sensitive (frame sub-matches before bare `frame N`; condition branch before the generic blob branch in renderCardBody:656 vs :676) and coalesceKey/category/lane assignments drive filtering and collapsing — the single parser table must encode the same precedence, with trace-cards.test.ts as the oracle.
4) Publishing surface: aiui-lowering-pipeline is a published package; the contract module makes channel trace vocabulary public API (semver pressure on every new stage label). Mitigation: export under a clearly-internal subpath/namespace and document instability; the alternative (a new private package) is illegal since published packages cannot depend on a never-published one.
5) The exhaustive switch must not sacrifice totality: `parseStageLabel` must be total ({t:\"unknown\"}), and the unknown arm must keep keying off TraceStage.kind exactly as trace-cards.ts:432-441 does today, or foreign/hand-made/future traces (TraceStageLike is deliberately all-optional, sources.ts:13-19) regress from degraded-render to broken-render.
6) Two dispatch layers interlock: RealtimeDiagnostic.kind (realtime.ts:106-110) currently leaks into labels via interpolation (intent-v1.ts:926-930), and the linter sidecar mints labels through a structural record callback (linter-sidecar.ts:78) — typing the label vocabulary without also typing that callback's stage parameter leaves a hole where new linter labels bypass the contract.
7) Adjacent but out of scope: the frame-log label ladder (frame-log.ts:111-175 → serve.ts narrator) is a parallel stringly wire with its own vocabulary ('hello', 'ack', 'push ${kind}', 'chunk ${kind}'); it does not cross into trace-ui, but the same contract pattern applies if it is ever consumed elsewhere — worth a note, not this change.
